# SiteGround cache & document TTFB audit

**Purpose:** Identify why document TTFB is ~1.33s on homepage and `/work/`, and what may prevent stronger full-page cache behavior for anonymous visitors.

**Scope:** Homepage (`/`, `/zh/`), Work page (`/work/`, `/zh/.../work`). No changes to portfolio AJAX; no new preload hints until current LCP preload is validated.

---

## 1. Caching layers relevant to homepage and /work/

| Layer | What it does | Controlled by |
|-------|----------------|---------------|
| **Dynamic Caching** | NGINX full-page cache (server memory). Serves cached HTML before PHP runs. Biggest TTFB win when HIT. | Speed Optimizer → Caching → Dynamic Caching ON. Server must support it (SiteGround hosting). |
| **File-Based Caching** | SG Optimizer writes/serves static HTML from `wp-content/cache/sgo-cache/`. Plugin sets `SG-F-Cache: HIT` / `MISS` / `BYPASS` on responses. | Speed Optimizer → Caching → File-Based Caching. |
| **Memcached** | Object cache for DB/meta (e.g. `get_terms`, ACF, options). Does not cache full HTML; speeds up PHP when generating a page. | Speed Optimizer → Caching → Memcached ON. |

For **anonymous** visitors, the main document (HTML) should be served from Dynamic Cache or File-Based Cache so TTFB drops from ~1.33s to well under ~200–300ms on cache HIT. If you see ~1.33s consistently, the HTML is likely not being served from either full-page cache.

---

## 2. What can prevent full-page cache for anonymous visitors

### 2.1 Speed Optimizer (SG Cachepress) – when the plugin bypasses or excludes

The plugin’s file cacher (`core/File_Cacher/`) only serves from cache when `is_cacheable()` is true. It sends **`SG-F-Cache: BYPASS`** when:

1. **Request is not GET**  
   Any non-GET request is not cached.

2. **URL is in “Exclude URLs from Caching”**  
   Option: `siteground_optimizer_excluded_urls`.  
   - **Check:** Speed Optimizer → Caching → “Exclude URLs from Caching”.  
   - **Ensure** `/` and `/work/` (and e.g. `*/work/`) are **not** listed.  
   - Excluded URLs are matched by regex; a pattern like `*work*` or `*/*` would bypass.

3. **Post type is in “Exclude Post Types from Caching”**  
   Option: `siteground_optimizer_post_types_exclude`.  
   - **Check:** Caching → “Exclude Post Types from Caching”.  
   - **Ensure** `page` is not excluded (homepage and Work are pages).

4. **Bypass cookies present**  
   If the request has any of these cookies (prefix match), cache is bypassed:  
   - `wordpress_logged_in_`  
   - `wpSGCacheBypass`  
   - `comment_author_`  
   - `woocommerce_items_in_cart`  
   - `edd_items_in_cart`  
   - **TranslatePress:** `trp_language` is **not** in the default list, so having that cookie does **not** by itself cause bypass.

5. **Bypass query parameters present**  
   Option: `sgo_bypass_query_params` (filter `sgo_bypass_query_params`).  
   - Default list in plugin includes e.g. `utm_*`, `_locale`.  
   - If the request has any of these query params, the page is not served from this cache layer.  
   - **Check:** Normal visits to `https://dev.vantage.pictures/` and `https://dev.vantage.pictures/work/` have no query string; if you use `?nocache=1` or similar, that could bypass if added to the list.

6. **Response sends “no-cache”–style headers**  
   The plugin checks `apache_response_headers()` for `cache-control: no-cache`.  
   - If any code sends `Cache-Control: no-cache` (or similar) before the file cacher runs, the page is treated as non-cacheable.  
   - **Theme/child:** No code in vantagepictures-child sends cache-control or nocache headers on the main document (only `portfolio-load-more.php` sends headers for **admin-ajax.php** responses).  
   - **Other plugins:** Possible; audit only – no plugin changes in this task.

7. **Content-Type not HTML**  
   Request must have `Accept: text/html`. Normal page loads do.

8. **AJAX / Cron**  
   `admin-ajax.php` and cron are never cached by the file cacher (and correctly show BYPASS). This does not affect the main document for `/` or `/work/`.

### 2.2 Logged-in users

By default, requests with `wordpress_logged_in_*` cookie bypass cache. So:

- **Anonymous** (no WordPress login cookie): should get cache.
- **Logged-in** (e.g. while testing): will bypass unless “Logged-in users cache” is enabled (separate cache per user; use with care).

For **testing cache**, use an incognito/private window and do **not** log in to WordPress on that profile.

### 2.3 Server-level (NGINX) Dynamic Cache

On SiteGround, Dynamic Caching is often implemented at the server (NGINX) level. The plugin enables/configures it; the actual serve happens before PHP.

- If Dynamic Caching is **OFF** in Speed Optimizer, the main document will always be generated by PHP → high TTFB.
- If **ON**, the server may still bypass (e.g. by cookie or URL) per SiteGround’s rules. Their docs or support can confirm which cookies/conditions cause NGINX to bypass.
- **Check:** In Speed Optimizer → Caching, confirm **Dynamic Caching** is **ON**. After a purge, open `/` and `/work/` in incognito and check response headers for the **main document** (first request in the waterfall). Look for headers like `X-Cache`, `X-Proxy-Cache`, `Cache-Status`, or similar (exact names depend on SiteGround’s NGINX setup). A HIT there would mean the document was served from Dynamic Cache.

## 3. How to confirm Dynamic Cache and Memcached are enabled and effective

### 3.1 In the WordPress admin

1. **Speed Optimizer → Caching**
   - **Dynamic Caching:** **ON**.
   - **File-Based Caching:** **ON** (so SG Optimizer can also serve/store HTML).
   - **Memcached:** **ON**.

2. **Memcached**
   - When ON, object cache is used for queries and options. You can’t “see” it in the UI except that repeat requests (and e.g. first AJAX portfolio load) can be faster. No code change needed; just confirm the toggle is ON.

3. **Exclusions**
   - **Exclude URLs from Caching:** Should **not** contain `/` or `/work/` (or patterns that match them).
   - **Exclude Post Types:** Should **not** exclude `page`.

### 3.2 In the browser (main document only)

1. Open **incognito** (no WordPress login).
2. Optional: Clear cache or use “Disable cache” in DevTools for the **first** request only.
3. Load `https://dev.vantage.pictures/` (or staging URL).
4. In DevTools → Network → select the **first request** (document, type `document`).
5. Check **Response Headers**:
   - **`SG-F-Cache`** (from SG Optimizer):
     - `HIT` = served from plugin’s file cache.
     - `MISS` = not in file cache (first request or expired).
     - `BYPASS` = request was not cacheable (see section 2).
   - Any **server** cache header (e.g. `X-Cache`, `X-Proxy-Cache`, `Cache-Status`) – if present, a HIT indicates Dynamic (NGINX) cache.

6. Repeat for `https://dev.vantage.pictures/work/`.

**Interpretation:**

- If the **main document** (not admin-ajax) shows **`SG-F-Cache: BYPASS`** in incognito with no query string and no login cookie, something in section 2 is triggering bypass (excluded URL, post type, cookie, or no-cache headers). Fix those first.
- If you see **`SG-F-Cache: MISS`** on first load and **`HIT`** on second load (same URL, incognito), file-based caching is working; TTFB on the second load should be much lower. If TTFB stays ~1.33s even on second load, then either the file cache is not being used to serve the response (e.g. NGINX serves first and doesn’t pass through) or another layer is slow.
- If the **main document** never shows HIT (only MISS or BYPASS), focus on making the request cacheable (exclusions, cookies, headers) and confirm both Dynamic and File-Based are ON.

## 4. Cookies, headers, and custom logic that could cause bypass

### 4.1 Cookies (theme/codebase)

- **Child theme:** Does not set any cookies. No bypass cookie is set by vantagepictures-child.
- **TranslatePress:** Can set a language cookie (e.g. when user uses the switcher). That cookie name is **not** in SG Optimizer’s default bypass list, so it should **not** cause bypass by itself.
- **SG Optimizer bypass list:** `wordpress_logged_in_`, `wpSGCacheBypass`, `comment_author_`, `woocommerce_items_in_cart`, `edd_items_in_cart`. If you ever need to force bypass for testing, you can set `wpSGCacheBypass` (plugin documents this).

### 4.2 Headers (theme/codebase)

- **Child theme:** No `header()` calls on the main document. Only `inc/portfolio-load-more.php` sends headers, and only for **admin-ajax.php** (portfolio AJAX), not for the main HTML request.
- So no theme code is sending `Cache-Control: no-cache` or similar for `/` or `/work/`.

### 4.3 Custom logic that might affect cache

- **TranslatePress:** Language is determined by URL (e.g. `/zh/`). Different URLs → different cache entries, which is correct. No need to exclude those URLs from cache.
- **Portfolio “no query on paged=1”:** The Work page intentionally skips the heavy portfolio query on first load and loads the first set via AJAX. That keeps the initial HTML light and does not send nocache headers for the document; it should not cause bypass.

## 5. GTM and third-party tag timing – what to delay

Third-party scripts (GTM, GA4, Facebook, LinkedIn, Clarity, Metricool, Bing, Snap) are loaded via **Google Tag Manager** (theme injects GTM at `wp_head` priority 1). They don’t need to run before first paint for the site to work.

**Recommendation:** In GTM, delay non-critical tags so they fire after the critical path (e.g. after LCP / window load). That reduces contention for network and main thread during the first ~2–3 seconds.

| Tag / purpose | Safe to delay? | Suggested trigger |
|---------------|----------------|-------------------|
| **GTM container** (loads first) | Keep as is (early in `<head>`) | Default – All Pages. |
| **GA4 page view** | Optional: can delay to Window Loaded if you accept slightly later first hit | All Pages or Window Loaded. |
| **Facebook (Meta)** | Yes | **Window Loaded** or **Timer 2–3s**. |
| **LinkedIn** | Yes | **Window Loaded** or **Timer 2–3s**. |
| **Clarity** | Yes | **Window Loaded** or **Timer 2–3s**. |
| **Metricool** | Yes | **Window Loaded** or **Timer 2–3s**. |
| **Bing (UET)** | Yes | **Window Loaded** or **Timer 2–3s**. |
| **Snap** | Yes | **Window Loaded** or **Timer 2–3s**. |

**How to do it in GTM:**

1. Open the tag (e.g. Facebook, LinkedIn, Clarity).
2. Change the trigger from “All Pages” (or “DOM Ready”) to:
   - **Window Loaded**, or  
   - A **Timer** trigger (e.g. 3 seconds after page load).
3. Save and publish. No theme or plugin code change.

**What to leave as-is:** GTM snippet in the theme (early in `<head>`). Optionally keep one early tag (e.g. GA4 page view) on “All Pages” if you want the earliest possible page view; all others can move to Window Loaded or Timer.

## 6. Checklist – document TTFB and cache

Use this to verify and fix:

- [ ] **Dynamic Caching** is **ON** (Speed Optimizer → Caching).
- [ ] **File-Based Caching** is **ON**.
- [ ] **Memcached** is **ON**.
- [ ] **Exclude URLs from Caching** does **not** include `/` or `/work/` (or patterns that match them).
- [ ] **Exclude Post Types** does **not** include `page`.
- [ ] Test in **incognito**, no WordPress login. Load `/` then `/work/`.
- [ ] For the **main document** request, check **Response Headers**: note `SG-F-Cache` (and any server cache header). If BYPASS, track down cause (section 2). If MISS then HIT on reload, file cache is working; if TTFB still high on HIT, investigate server/Dynamic Cache.
- [ ] In GTM, set non-critical tags (Facebook, LinkedIn, Clarity, Metricool, Bing, Snap) to **Window Loaded** or **Timer 2–3s**; keep GTM container load as is.

## 7. LCP preload – validation (no new preloads)

The current LCP preload (first hero image on front page and work page) is **implemented** but not yet validated. Do **not** add further preload hints until you:

1. Run a **fresh WebPageTest** (or equivalent) after the preload is live.
2. Confirm:
   - The **preloaded image** is the **LCP resource** (same URL as the element reported as LCP).
   - The **image request** starts **materially earlier** (e.g. in the waterfall it begins as soon as the preload is parsed, not only after body/hero is parsed).
   - **LCP** (and optionally FCP) **improves enough** to justify keeping the preload.

If validation shows no improvement or the preloaded image is not the LCP element, consider removing or narrowing the preload (e.g. only homepage or only work) and re-test. No other preloads should be added until this is confirmed.
